                                                                        # **Java **

                                                                        # Imports

                                                                        在Java文件中怎么避免 import java.utils.*包，以及import包顺序的问题

                                                                        当我们 Java 类中导入的某个包下类超过这里设置的指定个数，就会换成用*号来代替。

                                                                        当导入同一个包下的类的数量达到了设定的值后，就会用代替，例如，import java.utils.;

                                                                        Class count to use import with '': 默认是5，静态文件是2，可适当加大，加大到500，意指不建议引用这种方式，有些团队没有这个规范，可不设置

                                                                        当使用静态导入，导入同一个类下的静态方法的数量达到了设定的值后，就会用代替，例如，import static org.junit.Assert.;

                                                                        Names count to use static import with '*'：默认是5，可适当加大

                                                                        Packages to Use Import with'*'：下的包的排序即为import包的排序

                                                                        IDEA实时更新代码设置

                                                                        因为On frame deactivation，IDEA失去焦点的情况下会自动触发，而开发过程中，可能需要查询资料或与人聊天或干其它事，IDEA需要不停的失去焦点，每次失去焦点就自动触发设置的操作，设置成两个update后CPU一下子费很多，电脑瞬间变慢，

                                                                        所以没有必要这样，设置为Do nothing 最好，官方也默认的是 Do nothing。

                                                                        On 'Update' action选择Update classes and resources，顾名思义，就是更新类文件和资源文件，而Update resources只更新资源，如果是只更新资源文件的时候也可以用，但是考虑到通用设置的话，还是选择Update classes and resourcesDebug调试模式下（Java、JSP、JS、CSS、图片修改之后，点击Debug按钮，选择设置好的Update classes and resources就会生效，JS如果改动大的的话可能需要Ctrl + F5强刷页面或者清理页面缓存，否则无法生效（1、解决方案就是浏览器控制栏中Network勾选Disable cache（while DevTools is open），

                                                                        2、按F5或右键"重新加载"刷新页面内容或使用Ctrl + F5或Ctrl + Shift +R强制刷新页面，不使用浏览器缓存来刷新页面），如果是配置文件.xml之类的，可能无法及时生效无法及时生效操作：Redeploy（重新部署，发布到tomcat里，不重启tomcat，而是把原来的删掉，然后重新发布，比重启服务快），Redeploy无效的话，Restart server（重启服务，重启tomcat）

                                                                        优点：简单

                                                                        缺点：基于JVM提供的热加载仅支持方法块内代码修改，只有debug模式下，并且是在IDEA失去焦点时才会出发热加载，相对加载速度缓慢

                                                                        第2种：使用springloaded jar包

                                                                        a.下载jar包，github：https://github.com/spring-projects/spring-loaded

                                                                        b. 启动应用时添加VM启动参数：-javaagent:jar路径 -noverify

                                                                        优点：对Spring系列框架支持好（不含Spring boot）, 支持成员级别的修改（增删改方法、字段、注解），支持对枚举值集。

                                                                        缺点：与优点相对

                                                                        第3种：使用spring-boot-devtools提供的开发者工具

                                                                        spring-boot项目中引入如下依赖

                                                                        <dependency>

<groupId>org.springframework.boot</groupId>

<artifactId>spring-boot-devtools</artifactId>

</dependency>

                                                                        优点：简单，支持Spring-boot项目，支持成员级别的修改热部署。

                                                                        缺点：只支持spring-boot项目。

                                                                        第4种：使用Jrebel插件实现热部署(该插件14天免费试用)

                                                                        在线安装：菜单File -> Setting -> Plugin，点击右侧底部 Browse repositories，弹出框顶部输入：JReble for Intellij，选中安装即可。

                                                                        优点：强大，对各类框架支持，并且提供IDE插件的方式。

                                                                        在我们用 IntelliJ IDEA 向 SVN 或者 Git 提交代码的时候，IntelliJ IDEA 提供了一个自动分析代码的功能，即Perform code analysis：

                                                                        当我们勾选Perform code analysis之后，点击commit，IntelliJ IDEA 就会在提交代码之前对项目的代码进行分析检查，并将检查结果以错误和警告的形式展示出来：

                                                                        在这里，有一点需要着重说明，那就是：IntelliJ IDEA 的Code Analysis机制比较敏感，就算我们在文本注释中用错了标签或者注释时方法的参数与实际参数不一致，在Code Analysis的时候，都会以错误和警告的形式给出。因此我们常常会遇到这样的情况，就算代码中一点错误（提示）都没有，看起来没有飘红，当我们提交代码并进行Code Analysis的时候，仍然会收到一大堆的错误和警告提示，虽然这些错误和警告并不影响代码的运行。

                                                                        此外，在我们提交代码之前和之后都可以利用 IntelliJ IDEA 的自动化机制执行一些操作，例如勾选：

                                                                        在界面左侧空白区域邮件，勾选上 Annotate with Git Blame 即可显示对应文件详细代码提交记录，包括提交信息、提交人、变更文件等信息，非常强大！

                                                                        # 始终选择打开的文件

                                                                        选定了文件，即定位到了文件的位置

                                                                        这种方式有利有弊，用得久了，会打开很多文件，所以不勾选，使用下面按钮定位到具体位置即可

                                                                        # 全局设置

                                                                        Close Project->Costomize->All settings...

                                                                        可以进行全局共同的Maven、Compilier堆大小、各种插件设置

                                                                        # 项目级别设置

                                                                        • maven地址配置
                                                                        • Save Action配置
                                                                        • Compiler设置，默认700，适当加大到4096

                                                                        # IDEA JVM配置文件位置

                                                                        C:\Users\#{userName}\AppData\Roaming\JetBrains\IntelliJIdea#{version}\idea64.exe.vmoptions

                                                                        所以IDEA做过版本升级时，历史版本的配置文件夹是无效的，可以删除，减少C盘空间占用

                                                                        # 插件安装位置

                                                                        C:\Users\#{userName}\AppData\Local\JetBrains\#{version}\plugins

                                                                        所以IDEA做过版本升级时，历史版本的插件是无效的，可以删除，减少C盘空间占用
